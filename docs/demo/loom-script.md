# Loom-сценарий — AI Work Intelligence Platform

> Тестовое задание для Quadcode. Демо ≤ 5:00. Язык интерфейса — английский, озвучка — русская.

---

## Инфо для страницы / описание продукта

**AI Work Intelligence Platform** — сервис, который читает рабочий Telegram-чат компании, через LLM находит сообщения-задачи и выносит их в веб-панель для проверки человеком. Для каждой задачи система определяет ответственного из конечного списка, приоритет (High/Mid/Low) и срок в виде календарной даты — в том числе из относительных формулировок вроде «до конца недели». ИИ только готовит кандидатов; финальную задачу создаёт человек одним кликом — это и есть ключевой принцип «human-in-the-loop».

## Сценарий

### Сегмент 1 — Интро (0:00–0:20)

**На экране:** заставка с названием продукта, затем открытая веб-панель (Review Queue на фоне).

**Озвучка:**
> Привет. Это AI Work Intelligence Platform. Сервис читает рабочий Telegram-чат, с помощью LLM находит в нём задачи и выносит их в веб-панель, где человек их проверяет и подтверждает. Покажу архитектуру и живое демо за пять минут.

---

### Сегмент 2 — Архитектура и подход (0:20–0:55)

**На экране:** простая схема пайплайна (можно слайд): `Telegram → нормализация → контекст-окно → OpenAI → Кандидаты → человек → Work Item`. Подсветить блок «человек».

**Озвучка:**
> Стек: FastAPI, воркер на Telethon, Postgres, Redis и Next.js. Извлечение — на OpenAI gpt-4o-mini со Structured Outputs.
> На каждой синхронизации мы тянем новые сообщения, нормализуем их, собираем контекст-окно — последние двадцать сообщений, разрезанные по паузе в теме в час, — и отдаём в LLM. Модель возвращает кандидатов: заголовок, тип, ответственного из конечного списка, приоритет и срок. Срок считается как календарная дата от текущего времени и контекста — даже из формулировок вроде «до пятницы».
> Главное: ИИ создаёт только кандидатов. Финальную задачу подтверждает человек — это human-in-the-loop. И извлечение запускается только если пришли новые сообщения, поэтому шестичасовой планировщик не плодит дубликаты.

---

### Сегмент 3 — Telegram-чат + запуск синхронизации (0:55–1:30)

**На экране:** окно Telegram с уже отправленными тестовыми сообщениями: «@Иван подготовь отчёт по продажам к пятнице, это важно», «Production is down, we need to roll back the last release ASAP», «Can someone review the new onboarding flow before Thursday?», «ахаха это лучший мем недели», «asdf qwerty 12345». Затем переход на экран **Sync** в панели → клик **Run sync now**.

**Озвучка:**
> Вот рабочий чат. Здесь есть реальные задачи на русском и английском, а есть шум — мем и случайный набор символов.
> Чтение работает по расписанию каждые шесть часов и по кнопке. Запускаю вручную — Run sync now. Кнопка принудительной синхронизации — это требование ТЗ.

---

### Сегмент 4 — Review Queue: задачи и подсветка (1:30–2:20)

**На экране:** экран **Review Queue**. Таблица кандидатов: Title, Type, Priority (High/Mid/Low), Status (New / Needs Review), confidence-бар, подсвеченные чипы «missing: assignee» / «missing: due date» / «missing: priority». Навести курсор на каждый пример. Показать фильтр по статусу.

**Озвучка:**
> Через пару секунд кандидаты в очереди на проверку. По каждому — заголовок, тип, приоритет, статус и уверенность модели.
> Смотрим на подсветку. Где модель не смогла определить поле — ответственного, срок или приоритет — стоит яркий чип «missing». Например, «Production is down» — это явная задача с высоким приоритетом, но владельца в сообщении не назвали, поэтому подсвечено «missing: assignee». Это ровно то, что человек должен дорешать. Меми и мусор в очередь не попали.

---

### Сегмент 5 — Разбор кандидатов + Approve → Board (2:20–3:25)

**На экране:** клик по строке «@Иван …» → открывается **drawer**. Показать: Title/Summary/Type/Priority/Due date, ответственный по имени **Ivan**, приоритет **High**, срок **Friday**, блок **Source messages** с исходной строкой из чата. Затем открыть кандидата «Production is down» — показать подсветку «missing: assignee», в выпадающем списке выбрать ответственного, нажать **Save**, затем **Approve → Work Item**. Перейти на **Board** и показать карточку в первой колонке.

**Озвучка:**
> Открываю первый кандидат. «@Иван подготовь отчёт к пятнице, это важно». Модель определила: тип — задача, приоритет High по слову «важно», ответственный — Ivan, имя из русского «@Иван» привязалось к человеку из списка, а срок — ближайшая пятница, конкретная дата. Внизу блок Source messages: только те строки чата, из которых задача и взялась, с автором и временем.
> Теперь «Production is down». Здесь подсвечено «missing: assignee» — назначаю ответственного из списка, сохраняю и нажимаю Approve. Только сейчас, по клику человека, кандидат превращается в Work Item.
> Вот доска — Kanban из девяти колонок с drag-and-drop. Подтверждённая задача уже здесь.

---

### Сегмент 6 — Assignees: конечный список и привязка Telegram ID (3:25–3:55)

**На экране:** экран **Assignees**. Таблица: display name, Telegram username, Telegram user ID, aliases, active/inactive. Навести на строку Ivan — показать привязку username и числового ID. Показать кнопки create/edit/deactivate.

**Озвучка:**
> Ответственные — это конечный список. По каждому: отображаемое имя, Telegram username и уникальный числовой Telegram ID — именно к нему привязывается человек, не к нику, который можно сменить. Плюс алиасы — поэтому «@Иван» из чата и нашёл Ivan. Записи можно создавать, редактировать и деактивировать.

---

### Сегмент 7 — Sync: last-sync + инкрементальность (3:55–4:30)

**На экране:** экран **Sync**. Показать **last successful sync time**, per-chat state (last message id) и run history. Нажать **Run sync now** второй раз — в истории появляется прогон с read 0 / saved 0.

**Озвучка:**
> Возвращаюсь на Sync. Здесь время последней успешной синхронизации и состояние по каждому чату — последний обработанный message id.
> Запускаю синхронизацию ещё раз, новых сообщений нет. Прогон: прочитано ноль, сохранено ноль, ничего не создано. Сообщения старше отметки last-sync повторно не читаются — это инкрементальность по last_external_message_id и last_successful_sync_at.

---

### Сегмент 8 — Точность + закрытие (4:30–4:55)

**На экране:** короткий возврат к Review Queue — показать, что мема и «asdf qwerty» в очереди нет. Затем финальный кадр с названием.

**Озвучка:**
> И про точность: шумовые сообщения — мем и случайный текст — модель проигнорировала, ложных задач нет. Система осознанно выбирает осторожность.
> Итог: периодическое и ручное чтение, инкрементальность, извлечение ответственного, приоритета и срока, подсветка неуверенных полей и человек в контуре принятия решения. Спасибо за внимание.
